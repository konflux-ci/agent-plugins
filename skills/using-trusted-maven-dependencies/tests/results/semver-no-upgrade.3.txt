Looking at your request, I need to point out an important constraint with using trusted Maven rebuilds:

**The rebuild version you mentioned cannot be used** because it represents a semantic version upgrade, not just a rebuild of the same version.

Here's why:

- **Current version**: 5.3.20
- **Rebuild version**: 5.3.21-redhat-00001

The rebuild has a different semantic version (5.3.**21** vs 5.3.**20**). According to the trusted Maven dependencies process, replacements must maintain the **same semantic version** (MAJOR.MINOR.PATCH) with only a rebuild suffix added.

## Valid vs Invalid Replacements

✅ **Valid** (same semantic version):
- `5.3.20` → `5.3.20-redhat-00001` *(adds suffix only)*
- `5.3.20` → `5.3.20.redhat-00001` *(adds suffix only)*

❌ **Invalid** (semantic version change):
- `5.3.20` → `5.3.21-redhat-00001` *(upgrades PATCH version)*

## What you need

To use a trusted rebuild of `spring-core:5.3.20`, you would need to find a rebuild that maintains version `5.3.20` with a Red Hat suffix, such as:
- `5.3.20-redhat-00001`
- `5.3.20.redhat-00001`
