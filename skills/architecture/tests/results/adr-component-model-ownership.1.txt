## Konflux Component Model ADRs — Summary

### ADR-0056: Revised Component Model (`Implementable`)
**The central decision.** The old model required a new `Component` CR per use case (per branch, per release target), and API ownership was unclear after HAS was deprecated.

**Decided:**
- **Replace "Application" with "ComponentGroup"** as the grouping concept.
- **Introduce Component Versions** — a `Component` CR now holds multiple named versions (`spec.source.versions[]`), each representing a branch/tag. A single component can be referenced by multiple groups without re-registration.
- **Ownership split:** `build-service` owns the `Component` API and branch/version management. `integration-service` owns the new `ComponentGroup` API and will decommission the old `Application` API.
- **Default branch is no longer supported** — every version must have an explicit `revision`.
- **Phased rollout** ending in full decommission of the old Application/Component model.

---

### ADR-0060: ComponentGroups (`Proposed`)
**Defines the replacement for `Application`.** Companion to ADR-0056.

**Decided:**
- New CR `ComponentGroup` owned by integration-service, using `konflux-ci.dev/v1alpha1` API group.
- Relationship direction is **reversed**: ComponentGroup lists its components (top-down), not the other way around. This allows a Component to belong to multiple groups.
- Spec includes:
  - `spec.components[]` — list of `{name, componentVersion.name}` pairs
  - `spec.dependents[]` — other ComponentGroups that get snapshots triggered when this one does
  - `spec.testGraph` — optional DAG for **ordered/serialized test execution** (IntegrationTestScenarios with `onFail: run|skip`)
- Status tracks a `globalCandidateList` per group (last promoted image/commit per component).
- Release service must update `Release`, `ReleasePlan`, and `ReleasePlanAdmission` CRDs to support ComponentGroups alongside (or instead of) Applications.
- **Multi-version-per-component** is deferred; workaround is listing the same component twice with different versions.
- **Custom Snapshot creation** is a future enhancement.

---

### ADR-0029: Component Dependencies (`Accepted`)
**Predates the new model; defines nudging and PR Groups.**

**Decided:**
- `build-nudges-ref` field on `Component` CR declares that builds of one component should propagate digest updates to another (as automated PRs).
- **"Nudging" components** (those that nudge others) are never directly tested or released by integration-service — only promoted to the Global Candidate List.
- **"Nudged" components** proceed through the full test/promote/release cycle normally.
- **PR Groups**: PRs sharing a branch name prefix (delimited by `@`) are grouped and tested together by integration-service.
- Note: ADR-0056 says nudging will move to the integration test pipeline going forward.

---

### ADR-0004: Component Image Location (`Implemented`)
Early/foundational decision — each `Component` gets a dedicated Quay.io image repository automatically provisioned by image-controller. Not directly about the component model structure, but relevant background.

---

## Key Constraints for Your Proposal

1. **`build-service` owns `Component`; `integration-service` owns `ComponentGroup`** — proposals crossing those boundaries need justification.
2. **The old `Application` API is being decommissioned** — don't build on it.
3. **Default branch is explicitly unsupported** in the new model — every version needs an explicit `revision`.
4. **A component can belong to multiple ComponentGroups** — this is a core design goal; don't regress it.
5. **Custom Snapshot creation and multi-version-per-component are deliberately deferred** — if your proposal touches either, it may conflict with the design intent to solve those separately.
6. **API group is `konflux-ci.dev/v1alpha1`** for ComponentGroup (not the old `appstudio.redhat.com`).
