Here is a summary of all prior ADR decisions on the Konflux component model:

---

## Prior ADRs on the Konflux Component Model

### ADR 0004 — Component Image Location (`Implemented`)
**Decision:** Each `Component` CR gets a dedicated Quay.io image repository at `quay.io/redhat-user-workloads/<namespace>/<app>/<component>`, provisioned automatically by image-controller. No user-specified registry required at onboarding. This remains in effect.

---

### ADR 0029 — Component Dependencies (`Accepted`)
**Decision:** Introduced `build-nudges-ref` on the `Component` spec — a list of other Components whose image digests should be auto-updated (via PR) when a build of the source component completes. Also introduced **PR Groups** (via `@`-prefixed branch naming) for coordinated cross-component pre-merge testing. Key behavioral rules:
- **Nudging** components skip tests, skip deployment/release, but promote to the Global Candidate List.
- **Nudged** components run tests and deployments normally.
- `build-service` files the digest-update PRs; `integration-service` handles testing logic.

---

### ADR 0056 — Revised Component Model (`Implementable`)
**Decision:** The biggest architectural shift. Replaces the `Application` CR concept with **Component Versions** and **Component Groups**:
- A `Component` now has `spec.source.versions[]` — a list of named branches/tags, each with its own build pipeline config. One `Component` CR covers all release branches.
- **build-service** takes sole ownership of the `Component` API.
- **integration-service** owns a new `ComponentGroup` API (replaces `Application`).
- The old Application/Component model is to be decommissioned in a phased migration.
- Nudging moves to integration test pipelines.

---

### ADR 0060 — ComponentGroups (`Proposed`)
**Decision:** Deprecates and replaces the `Application` CR with a `ComponentGroup` CR owned by integration-service. Key design choices:
- Relationship defined **top-down**: the `ComponentGroup` lists which `Component`/`componentVersion` pairs it contains (inverts the old `spec.application` field on `Component`).
- A single Component can belong to **multiple** ComponentGroups (solving the old 1:1 constraint).
- The `spec.testGraph` field enables **ordered integration testing** (DAG of `IntegrationTestScenario` dependencies).
- The `spec.dependents` field triggers Snapshot creation for child ComponentGroups when a parent Snapshot is created.
- The `status.globalCandidateList` replaces the per-Application GCL.
- Release CRDs must be updated to reference a `ComponentGroup` instead of an `Application`.

---

### ADR 0067 — Nudging Relationship via Singleton CRD (`Proposed`, partially supersedes ADR 0029)
**Decision:** Moves nudging relationship storage off the `Component` CR into a new **`NudgeConfig`** singleton CRD per namespace, owned by integration-service. Key points:
- At most one `NudgeConfig` per namespace, enforced by CEL validation on `metadata.name`.
- Each entry specifies `from`, `to`, and `mode` (`immediate` or `validated`).
- **Immediate mode:** nudge fires as soon as the source build PipelineRun succeeds (preserves ADR 29 behavior).
- **Validated mode:** nudge fires only after the Snapshot for the named `gatingGroup` passes integration tests — eliminates redundant downstream builds.
- build-service retains Renovate tooling; integration-service becomes the nudge orchestrator.
- Cycle detection, self-nudge prevention, and duplicate-pair rejection enforced via CRD-level CEL rules.
- One-time automated migration from `build-nudges-ref` → `NudgeConfig` (mode: immediate).
- The `build-nudges-ref` field on `Component` is deprecated and will be removed.

---

## Summary for Your Proposal

The key active decisions you must be aware of when proposing changes to the component model are:

| Topic | Current ADR decision |
|---|---|
| What is a Component? | One CR per git repo, with multiple named `versions` (branches/tags) inside it — **ADR 56** |
| What replaces Application? | `ComponentGroup` CR, owned by integration-service — **ADR 60** |
| Who owns the Component API? | build-service — **ADR 56** |
| Who owns ComponentGroup API? | integration-service — **ADR 60** |
| How are nudge dependencies stored? | New `NudgeConfig` singleton CRD per namespace — **ADR 67** |
| Who orchestrates nudges? | integration-service (moved from build-service) — **ADR 67** |
| Cross-component scope for nudges? | Namespace-scoped (widened from Application-scoped) — **ADR 67** |

All four ADRs are interlinked — any new proposal touching the component model should cite ADR 56 and 60 as the baseline, and be careful not to conflict with the ownership boundaries (build-service owns `Component`, integration-service owns `ComponentGroup` and `NudgeConfig`).
