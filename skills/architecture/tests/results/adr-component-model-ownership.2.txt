## ADR Summary: Konflux Component Model

### ADR-0056 — Revised Component Model (`status: Implementable`)

This is the central decision. The existing model — one Component per use case, all coupled to an Application — was found to create unnecessary overhead and unclear API ownership (HAS was deprecated).

**What was decided:**

- **Replace "Application" with "Component Group"** as the grouping primitive.
- **Introduce Component Versions**: a Component can now have multiple named versions, each tied to a specific branch or tag. A single component can be reused across multiple groups without re-registration.
- **Ownership split**:
  - `build-service` owns the `Component` API and manages versions/branches.
  - `integration-service` owns the new `ComponentGroup` API and will decommission the old `Application` API.
- The new `Component` spec nests versions under `spec.source.versions[]`, each with its own `revision`, `context`, `dockerfileUri`, `skip-builds`, and `build-pipeline` overrides.
- **Implementation is phased**: new model ships as tech preview, then old model is auto-migrated and decommissioned.
- Nudging moves to integration test pipelines.

---

### ADR-0060 — ComponentGroups (`status: Proposed`)

Companion to ADR-0056, detailing the `integration-service` side.

**What was decided:**

- Introduce a new `ComponentGroup` CR to replace `Application`.
- Relationship is now **top-down**: the `ComponentGroup` lists which Component+version combinations it contains (instead of Components pointing up to an Application). This allows a Component to belong to multiple groups.
- `spec.components[]` lists Component + `componentVersion.name` pairs.
- `spec.testGraph` provides an optional DAG for **test ordering** (run expensive/dependent tests in sequence).
- `spec.dependents` allows one group's new snapshot to trigger snapshot creation for other groups.
- `status.globalCandidateList` is owned by integration-service and tracks the last promoted image per component.
- The `Release`/`ReleasePlan`/`ReleasePlanAdmission` CRDs need minor updates to support `ComponentGroup` alongside or instead of `Application`.
- Multiple versions per component in one group is intentionally **deferred** to a future "custom Snapshot creation" feature.

---

### ADR-0029 — Component Dependencies / Nudging (`status: Accepted`)

An earlier (2023) decision that introduced the **nudging** mechanism, still relevant to the new model.

**What was decided:**

- New `build-nudges-ref` field on `Component` lets one component declare it should push digest updates to another (forming a DAG).
- **Nudging** components: builds skip all testing/promotion; build-service propagates a PR with the new digest to the nudged repo.
- **Nudged** components: tested and promoted normally.
- **PR Groups**: PRs sharing a branch-name prefix (delimited by `@`) are treated as a group and tested together by integration-service.
- ADR-0056 later moved nudging into the integration test pipeline.

---

### ADR-0004 — Component Image Location (`status: Implemented`)

An early (2022) decision, now largely superseded.

**What was decided:** Each Component gets an auto-provisioned Quay.io image repository under `quay.io/redhat-user-workloads/<namespace>/<app>/<component>`, managed by `image-controller` with scoped robot account tokens.

---

## Key Takeaways for Your Proposal

Before proposing changes, be aware:

1. The community has **already decided** to deprecate the `Application` CR in favor of `ComponentGroup` (ADR-0060). Any proposal that touches this boundary needs to align with or explicitly engage that decision.
2. Multi-version support is now **first-class** in the `Component` spec (ADR-0056) — you don't need a new Component per branch.
3. Service ownership is clearly split: `build-service` → Component/versions, `integration-service` → ComponentGroup.
4. The `Snapshot` resource remains a shared coupling point between integration-service and release-service — noted as a known limitation.
5. ADR-0056 is `Implementable` (in progress); ADR-0060 is still `Proposed` — so the ComponentGroup side has room for input.
