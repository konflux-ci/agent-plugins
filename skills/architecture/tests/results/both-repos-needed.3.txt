Based on the architecture guidance above, here's what you should review to give informed feedback on a release strategy proposal:

## Key Areas to Review

### 1. **Release Service Architecture** (Primary)
Start with `architecture/core/release-service.md` in the [konflux-ci/architecture](https://github.com/konflux-ci/architecture) repo to understand:
- Current release service responsibilities and boundaries
- What release strategies already exist and how they work
- CRDs and APIs the release service owns
- How it integrates with other services (pipeline, integration, etc.)

### 2. **Existing Release Strategy ADRs**
Search the `ADR/` directory for decisions about release strategies. Look for:
- ADRs on release pipeline design and strategy patterns
- Prior decisions about what makes a valid strategy
- Constraints or principles that might affect the new strategy

### 3. **User-Facing Release Documentation**
Check the [konflux-ci/docs](https://github.com/konflux-ci/docs) repo, specifically:
- `modules/releasing/` - How release strategies are currently documented and used
- Current workflow examples showing existing release strategies
- User configuration and best practices for releases

### 4. **Integration Points**
Evaluate how the new strategy interacts with:
- The pipeline service (how strategies are invoked)
- The integration service (testing requirements)
- Enterprise contract and policy enforcement
- Multi-tenant and RBAC considerations (if applicable)

### 5. **Key Questions to Ask**
- Does this strategy fit within existing release service responsibilities, or does it require new ones?
- Does it conflict with or duplicate existing ADRs or strategies?
- What's the user-facing contract (CRDs, configuration) and is it consistent with existing patterns?
- How does it handle enterprise contract validation and compliance requirements?
- Are there operational/troubleshooting implications not yet documented?

Start with the architecture repo's overview and the release-service spec, then dig into specific ADRs and docs as needed.
