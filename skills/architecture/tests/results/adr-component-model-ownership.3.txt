## ADRs on the Konflux Component Model

### ADR-0056 — Revised Component Model (`Implementable`)
**The core decision.** The old `Application`/`Component` model is being replaced with two new concepts:

- **Component Versions** (`spec.source.versions[]`): A single `Component` CR now carries multiple branches/tags as named versions. No more creating a separate Component per branch/release stream. `build-service` owns this API going forward.
- **Component Groups** (replaces `Application`): A `ComponentGroup` CR references one or more `(Component, version)` pairs and forms a DAG representing the release structure. A component version can belong to multiple groups. `integration-service` owns this API and is responsible for decommissioning the old `Application` API.

The old model forced re-registration of a component for each use case and had unclear ownership (originally HAS, which is now deprecated). The new model fixes both.

**Key spec changes to `Component`:**
- `spec.source.versions[]` — list of named versions (branch + optional dockerfile/context/pipeline overrides)
- `spec.default-build-pipeline` — default pipeline for all versions, overridable per version
- `spec.actions` — imperative triggers (e.g., `trigger-push-build`, `create-pipeline-configuration-pr`)
- `spec.containerImage` — set by controller when an `ImageRepository` is created

---

### ADR-0060 — ComponentGroups (`Proposed`)
**The integration-service's design for the replacement of `Application`.** Key decisions:

- `ComponentGroup` lists `(component, componentVersion.name)` pairs top-down (instead of components pointing up to an application). This allows a component to belong to multiple groups.
- Adds a `testGraph` field to define an ordered DAG of `IntegrationTestScenarios` — enabling test serialization and avoiding running expensive tests unnecessarily.
- Adds a `dependents` field: when a snapshot is created for group A, snapshots are also created for listed dependent groups.
- `status.globalCandidateList` moves onto the `ComponentGroup` (away from `Application`).
- The `Release` CRD's `spec.application` field will be updated to support `ComponentGroups`.
- Multiple component versions per component in one group is **deferred** until custom Snapshot creation is implemented.
- API group is `konflux-ci.dev/v1alpha1` (not the old `appstudio.redhat.com`).

---

### ADR-0029 — Component Dependencies (`Accepted`)
**Decided how components express build-time dependencies (nudging).** Key decisions:

- `build-nudges-ref` field on `Component` CRs declares that builds of component A should propagate digest updates to component B as a PR ("nudging").
- `integration-service` skips all testing and promotion for **nudging** components; only the **nudged** component runs tests and triggers releases.
- **PR Groups**: PRs sharing the same `@`-prefixed source branch name are treated as a group, and integration-service tests them together.
- `build-service` acts like a renovatebot controller: on a successful nudging build, it files/updates a PR on the nudged component's repo with the new digest.

> **Note:** ADR-0056 mentions that nudging will eventually move into the integration test pipeline, superseding some of this.

---

### ADR-0004 — Component Image Location (`Implemented`)
An earlier, fully implemented decision: each `Component` gets its own auto-provisioned Quay.io image repository under `quay.io/redhat-user-workloads/<namespace>/<app>/<component>`, with a scoped robot account token. Users don't need to supply an image registry.

---

## Bottom Line for Your Proposal

Before writing your proposal, note these constraints already decided:
1. The `Application` CR is being **deprecated** — any proposal should use or extend `ComponentGroup`, not `Application`.
2. `build-service` owns the `Component` API; `integration-service` owns `ComponentGroup`. Proposals touching both need to respect this ownership split.
3. Component versions must be **named branches/tags** — default-branch-only builds are explicitly not supported in the new model.
4. Nudging (`build-nudges-ref`) is Accepted but may be relocated per ADR-0056's roadmap.
5. ADR-0060 is still `Proposed` — that's the most open area for influence right now.
