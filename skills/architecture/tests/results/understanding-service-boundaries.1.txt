## Finding Integration vs Release Service Responsibilities

Go to the [konflux-ci/architecture](https://github.com/konflux-ci/architecture) repository and read:

1. **`architecture/core/integration-service.md`** - Documents what the integration service owns and is responsible for
2. **`architecture/core/release-service.md`** - Documents what the release service owns and is responsible for

Each service spec covers:
- **Responsibilities** - what the service does
- **CRDs owned** - which Kubernetes custom resources the service manages
- **Interactions with other services** - how it communicates with build, release, pipeline, etc.

## Quick Context

In Konflux:
- **Integration Service** typically handles the testing/integration phase - running tests, validating builds, managing integration test pipelines
- **Release Service** typically handles the release phase - creating release artifacts, managing release pipelines, deploying to production

But for the authoritative definitions specific to your version of Konflux, read those service docs directly.

## If You Need Design Rationale

If you're trying to understand **why** responsibilities are divided this way, check the `ADR/` directory in the same repo for any Architecture Decision Records related to service boundaries or the integration/release workflow.
